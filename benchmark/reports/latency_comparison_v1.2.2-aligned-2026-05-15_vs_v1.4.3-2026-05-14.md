## 핵심 한 줄 결론

**모집단을 통일하니 recall total 격차의 약 65–70% 가 사라졌다.** 이전 비교(5/12 vs 5/14) 에서 "v1.4.3 가 recall 에서 3.3배 빠름" 으로 보였던 격차는 동일 모집단(20건) 정렬 후 **1.16배** 로 축소. 이전 비율의 **약 72%(= 1 − 1.16/3.30) 가 모집단 차이의 기여분**이었고, **나머지 약 0.16배만이 SDK · eval_mode · index_type · 엔드포인트 4 변수의 합산 기여분**.

phase 분해로 더 자세히 보면, **`score` 단계는 두 환경이 거의 동일하거나 v1.2.2 flat 이 오히려 빠르다** (recall T5 154.6 ms vs 135.8 ms = 1.14×; capture T1 73.7 ms vs 169.5 ms = **v1.2.2 가 2.3 배 빠름**). 이는 작은 모집단(≤20건)에서는 IVF_VCT 의 클러스터 탐색 오버헤드가 flat brute-force 보다 무겁다는 신호. 본질적 격차는 **`vault_topk` 단독 (recall T5 75.9 vs 15.9 = 4.77×)** 에서만 유지 — eval_mode `rmp`→`mm32` 와 vault payload 형식 차이가 만드는 phase.

`searchable` / `multi_capture` 는 측정 메커니즘 차이(v1.2.2 client polling vs v1.4.3 server-push `MERGED_SAVED`) 가 여전히 결과를 지배. 이번 측정에서는 v1.4.3 의 5/14 측정의 `merge_wait` 가 p50/p95 모두 15.6 s 수준으로 측정되어 → v1.2.2 aligned 가 14–22배 더 "빠르게" 보이지만 §5 참조.

```
v1.4.3          █████████████████████████  256ms              (1.16배 빠름)
```

**가장 큰 phase 격차였던 `vault_topk` 의 4.77 배는 이번 정렬 측정에서도 유지된다.** 모집단·reset 정책 무관 — eval_mode rmp→mm32 와 vault 의 decrypt payload 형식 차이가 본질적 격차의 원천. 이번 측정으로 그것이 "유일한 본질적 phase 격차" 임이 확정.

## 4. score phase 의 N-선형성 — flat vs ivf_vct 교차점

> **[발견]** 작은 모집단 (≤20건) 에서는 **flat brute-force 가 ivf_vct(nprobe=6) 보다 빠르다**.

| 측정 | 인덱스 모집단 (recall 시점 또는 capture 시점) | score p50 (ms) | 알고리즘 |
| ---- | --------------------------------------------- | -------------- | -------- |
| v1.2.2 5/12 (T5 recall) | 42 건 | 509.7 | flat (N-선형) |
| v1.2.2 5/15 aligned (T5 recall) | 20 건 | **154.6** | flat (N-선형) |
| v1.4.3 5/14 (T5 recall) | 20 건 | 135.8 | ivf_vct(nprobe=6) |
| v1.2.2 5/15 aligned (T1 capture) | 0 건 | **73.7** | flat (N-선형) |
| v1.4.3 5/14 (T1 capture) | 0 건 | 169.5 | ivf_vct(nprobe=6) |

**관찰**:

1. **flat 은 N 에 선형**: 42 → 20 (모집단 0.48 배) → score 509.7 → 154.6 (0.30 배). 모집단 비율보다 더 큰 감소 — 클러스터 측 부하 변동도 영향. 0 → 20 (T1 capture 0 vs T5 recall 20) → 73.7 → 154.6 (2.1 배 증가). 모집단이 0 에서 20 으로 늘 때 score 가 약 80 ms 증가하므로, **20 건당 ~80 ms** 의 flat 비용 추정.
2. **ivf_vct 는 N 에 거의 무관**: 0 건 (capture) 169.5 ms, 20 건 (recall) 135.8 ms — 같은 데이터셋(0–20) 에서 ms 단위 차이가 없음. nprobe=6 의 고정 비용이 약 130–170 ms 수준.
3. **교차점**: 두 알고리즘의 score 비용이 같아지는 모집단 N. 0 건일 때 flat 74, ivf_vct 170 → 약 96 ms 격차. flat 의 N 당 비용을 80 ms / 20 건 = 4 ms/건 으로 단순 외삽하면, **N ≈ 24 건 부근에서 교차** (74 + 4×24 = 170). 이는 작은 모집단 사용처 (벤치마크 / 초기 테넌트 / soft-launch) 에서는 flat 이 알고리즘 측면에서 유리하다는 추정 근거.

> **[운영 함의]** 본 비교의 4 변수 격차가 작은 모집단에서 거의 사라지므로, 모집단이 100 건 미만으로 유지되는 환경 (예: 신규 사용자, 데모 인덱스) 에서는 **SDK 업그레이드 단독의 latency 이득이 미미**. 이미 100 건 이상이거나 빠르게 늘어나는 모집단에서만 ivf_vct 의 N-무관 속성이 가치를 만든다. capture 의 vault_topk 만은 모집단 무관으로 빨라지므로 모집단 무관하게 v1.4.3 도입의 단일 가치 — `vault_topk` 4.77 배 가속.

## 5. ⚠ Searchable / Multi_capture — 측정 메커니즘 차이 (caveat 유지)

§3·§4 의 결론과 별개로, 이 두 feature 는 **두 환경의 측정값이 가리키는 시점 정의 자체가 다름**:

| 측정값 (T10 short_en_searchable) | v1.2.2 aligned (5/15)                              | v1.4.3 (5/14)                                      |
| -------------------------------- | -------------------------------------------------- | -------------------------------------------------- |
| total p50                        | **772.4 ms**                                       | **16391.6 ms** (21.2 배 느림) ⚠                     |
| 측정 메커니즘                    | 클라이언트 폴링 (200 ms granularity, top-1 cos ≥ 0.999) | 서버 push (`await_searchable=True` → MERGED_SAVED) |
| 지배 phase                       | `insert_searchable` 609 ms (폴링 사이클 포함)      | `merge_wait` 15694 ms — total 의 95.7%             |

- v1.2.2 폴링: client 가 score → vault decrypt 사이클을 200 ms 간격으로 반복해 top-1 cos ≥ 0.999 도달 시점을 측정. **raw shard 가시화 시점만 측정**.
- v1.4.3 server-push: 클러스터가 raw → non-raw shard 이동 (`MERGED_SAVED`) 완료를 RPC 반환으로 통지. **클러스터 측 merge 워커 처리 완료 시점**까지 포함.
- 5/14 v1.4.3 의 `merge_wait` 가 p50/p95 모두 15.6 s 수준 (n=8 의 분포가 좁게 모여 있음) 으로 측정된 것은 그 시점의 클러스터 측 merge 워커 처리 소요 시간을 그대로 반영한 값 (5/14 리포트 §1 참조 — `async_merge_by_request_ids` → `MERGED_SAVED` 전이가 15.5–15.7 s). v1.2.2 측 polling 측정은 이 단계를 보지 않으므로 "21–22 배 빠름" 으로 보이지만 **두 measurement 가 다른 시점을 가리킨다는 사실** 자체가 변하지 않음.

**multi_capture (T13–T14) 도 동일 구조**: v1.4.3 의 `insert_batch` 는 `await_completion=True, execute_until="segmentation"` 로 MERGED_SAVED 까지 대기 — 같은 cluster 측 merge 단계를 측정값에 포함. v1.2.2 의 batch insert 는 SDK 가 내부적으로 N×single 로 직렬화한 wall-clock 만 측정.

**→ §1 표의 21.2× / 22.3× / 22.6× / 11.6× / 8.9× 비율을 외부 인용할 때는 반드시 "측정 메커니즘이 달라 직접 비교 부적절" 단서를 함께 표기.** §8 #4 (v1.4.3 측에 polling 측정 추가) 후속 작업 전까지 이 caveat 은 유지된다.

## 7. 한 줄 결론들

1. **§8 #1 의 답**: "v1.4.3 가 recall 에서 3.3 배 빠름" 중 (모집단 42 vs 20) 의 기여분이 **약 2.85 배**, (SDK + eval_mode + index_type + 엔드포인트) 4 변수 합산 기여분이 **약 1.16 배**. 4 변수 단독 격차 중에서도 **`vault_topk` 한 phase (4.77 배) 가 단독 주역** — 나머지 phase 들은 거의 비등하거나 v1.2.2 가 빠르다.
2. **알고리즘 측면 결론**: 작은 모집단 (≤약 20–24 건) 에서는 ivf_vct(nprobe=6) 가 flat 대비 score phase 측면에서 우위 없음 — 오히려 빈 인덱스에서는 flat 이 2.3 배 빠름. **v1.4.3 의 알고리즘적 이득은 N 이 충분히 클 때만 가치 발생**.
3. **production 가이드**: 모집단이 100 건 미만으로 유지되는 워크로드는 SDK 업그레이드 단독의 latency 이득이 미미. capture 의 `vault_topk` (모집단 무관 2–5 배 빠름) 가 유일한 본질적 이득.
4. **searchable/multi_capture 인용 시**: 본 비교에서도 시점 정의 차이가 유지됨. 외부 인용 시 (a) 측정일 (5/15 v1.2.2-aligned / 5/14 v1.4.3), (b) 두 측정의 시점 정의 차이 (raw shard 가시화 vs MERGED_SAVED) 를 함께 표기.
5. **5/12 production-누적 측정과의 차이 (방법론)**: 이번 aligned 측정은 매 시나리오 빈 인덱스에서 시작. capture/searchable/multi 측정값이 5/12 대비 30–60% 감소한 부분은 production 인덱스의 누적 데이터가 만든 부풀림 효과였음 — 향후 "v1.2.2 baseline" 으로 인용할 때는 5/12 가 아닌 **5/15 aligned 측정**을 권장.

## 8. 후속 조사 제안

1. **모집단 N=200 / N=2,000 추가 측정** — 본 측정은 N=20 단일 점. flat 의 N-선형 비용 곡선 vs ivf_vct 의 N-무관 비용을 더 넓은 N 범위에서 측정해 score phase 의 cross-over 모집단을 실측. 러너의 `_prime_bench_index(n_records=N)` 호출만 바꿔 측정 가능. CLI flag `--prime-records` 추가하면 1-line 변경.
2. **`vault_topk` 4.77 배 격차의 wire-level 분해** — gRPC interceptor 로 (a) envector→vault 로 흐르는 ciphertext 바이트 크기 (rmp vs mm32), (b) vault 의 decrypt CPU 시간 분리 측정. eval_mode 와 vault 인프라 기여분 분리.
3. **두 환경의 vault 엔드포인트 교차 측정** — 가능하다면 v1.2.2 측을 v1.4.3 의 vault endpoint (`161.118.149.143`) 로, 또는 그 반대로 임시 전환해서 측정. 인프라 단독 기여분 추가 분리.
4. **searchable 측정 시점 정렬** — v1.4.3 측에 client-side cosine 폴링 wrapper 추가, 양쪽을 같은 "raw shard 가시화" 시점으로 재측정. searchable 비교에서 (시점 정의) 와 (클러스터 merge 처리량) 분리. 직전 비교 §8 #4 와 동일 제안.
5. **v1.4.3 `merge_wait` 의 시점별 재측정** — 별도 시점에 v1.4.3 단독 재측정해 5/14 의 16 s 수준 측정값이 그 시점에만 관찰되는 값인지 다른 시점에서도 같은 수준인지 확인. 시점에 따라 변동한다면 searchable/multi_capture 의 8–22 배 격차도 그에 따라 달라질 가능성.
