# enVector/Rune 경쟁 포지셔닝 프레임워크

## 목적

"멀티에이전트 집단지성이 비즈니스에서 보편화될 것인가?"라는 질문에 대한 체계적 방어 논리 프레임워크. 특정 과제용이 아니라 어떤 자리에서든 재사용 가능한 논거 세트.

## 주요 대상

공공/정부 과제 심사위원 (시니어급). "에이전트는 개인 장난감이지, 기업 의사결정 도구가 아니다"라는 인식에 대응.

## 기존 전략 맥락

- 국방/치안/금융에서 태핑 사례는 있으나, "이것이 미래다"라는 체계적 답변이 부재
- 공공 수요 자체는 어느 정도 설득 완료

## 4단 프레임: 패턴 - 전환점 - 가속기 - 아키텍처

### 1단: 패턴 인식 — "모든 개인 도구는 공유 메모리를 얻으면 기업 인프라가 되었다"

#### 명제

개인 도구가 기업 인프라로 전환되는 것은 IT 역사에서 반복된 구조적 패턴(IT Consumerization)이며, 전환의 trigger는 항상 "조직 수준의 공유 메모리"였다.

#### 학술적 근거

"IT Consumerization"은 2001년 Leading Edge Forum의 Douglas Neal이 명명한 개념으로, MIS Quarterly, AMCIS 등 주요 IS 학술지에서 검증된 프레임워크.

핵심 메커니즘: 개인이 익숙한 도구를 업무에 가져오고 → 부서 단위로 퍼지고 → IT 부서가 뒤늦게 공식 채택하는 bottom-up 패턴. 예외가 아니라 반복되는 구조적 현상.

#### 역사적 증거

| 시대 | 개인 도구 | 공유 메모리가 생긴 순간 | 기업 인프라로 전환 |
|------|----------|----------------------|-------------------|
| 1990s | 개인 이메일 | 공유 주소록 + 메일링리스트 + 아카이브 | Exchange/Outlook → 기업 커뮤니케이션 표준 |
| 2000s | 개인 스프레드시트 | Google Sheets 동시편집 + 공유 드라이브 | 기업 데이터 협업 표준 |
| 2010s | 개인 메신저 | 채널 + 검색 가능한 히스토리 | Slack/Teams → 기업 지식 허브 |
| 2010s | 개인 클라우드 서버 | IAM + 공유 인프라 관리 | AWS/Azure → 기업 IT 인프라 표준 |
| 2020s | 개인 AI 에이전트 | **암호화된 조직 메모리** | ? → 기업 AI 의사결정 인프라 |

공통 패턴: 개인 도구 자체는 "편하다"는 수준이지만, 조직 수준의 공유 메모리가 붙는 순간 "없으면 일이 안 되는" 인프라로 격상.

대표 사례 — Slack: 게임회사 내부 IRC → 개발팀 자발 채택 → 2년 만에 170만 DAU → IBM 전사 도입 → $27.7B Salesforce 인수. 전환점은 "채널 + 검색 가능한 아카이브"라는 공유 메모리.

대표 사례 — Dropbox: 직원이 개인 계정으로 업무 파일 공유 → IT가 차단 시도 → 이미 업무 의존 → Dropbox Business로 공식화.

#### 핵심 메시지

> "AI 에이전트가 기업 도구가 될 것이냐는 질문은, 2005년에 '이메일이 기업 도구가 될 것이냐'고 묻는 것과 같다. 질문은 '될까?'가 아니라 '언제?'이며, 역사적으로 전환의 trigger는 항상 공유 메모리였다."

#### 참고 문헌

- Harris, Ives et al. "IT Consumerization: When Gadgets Turn Into Enterprise IT Tools" (MIS Quarterly Executive, 2012)
- Junglas et al. "When Consumer IT Turns Into Enterprise IT: Two IT Consumerization Models" (FSU Research)
- IT Consumerization - A Theory and Practice Review (AMCIS 2012)

### 2단: 전환점 — "빅테크가 이미 베팅했다"

#### 명제

"에이전트에 공유 메모리가 필요하다"는 가설이 아니라, 빅테크가 동시에 수십억 달러를 베팅한 시장 신호다.

#### 빅테크 에이전트 메모리 출시 타임라인

| 시기 | 빅테크 | 움직임 | 가격 |
|------|--------|--------|------|
| 2024 Q4 | Anthropic | Claude Team/Enterprise — Projects, Activity Feed | $25-150/user/month |
| 2025 Q1 | Microsoft | Copilot Memory — Semantic Index | $54-75/user/month |
| 2025 Q2 | OpenAI | ChatGPT Enterprise — 공유 GPTs, 프로젝트 공유 | $25+/user/month |
| 2025 H2 | Google | Gemini + NotebookLM — 공유 노트북 | $14-34/user/month |

#### 애널리스트 데이터

- **Gartner**: 기업 앱의 40%가 2026년까지 task-specific AI 에이전트 내장 (2025년 5% 미만에서)
- **Gartner**: 기업의 70%가 2029년까지 IT 인프라 운영에 agentic AI 배포
- **Gartner**: "Multi-agent systems"를 2026년 top strategic technology trend로 선정
- **McKinsey**: 2025년 기준 기업의 23%가 agentic AI를 최소 1개 사업부에서 스케일링, 39% 실험 중
- **Forrester**: 2028년까지 일상 업무 의사결정의 15%가 agentic AI에 의해 자율 수행 (2024년 0%)
- **Forrester**: 2026년 기업 앱은 "디지털 도구 제공" → "AI 에이전트라는 디지털 인력 수용" 단계로 전환
- 전세계 AI 지출 2026년 $2.52T 예상 (YoY 44% 증가)

#### 빅테크 솔루션의 구조적 한계

- 각 벤더의 에이전트에 종속 (Claude 메모리는 Claude에서만, Copilot은 MS 생태계에서만)
- Plaintext — 서버가 모든 내용을 열람 가능
- 벤더 사일로이지, 조직 인프라가 아님

#### 핵심 메시지

> "빅테크가 '에이전트에 공유 메모리가 필요하다'는 것을 증명해줬다. 하지만 그들의 솔루션은 각자의 에이전트에 갇힌 사일로다. 진짜 기업 인프라는 벤더를 넘나드는 공유 메모리여야 하고, 그 gap이 enVector/Rune의 자리."

### 3단: 가속기 — "한국에서는 규제가 멀티에이전트를 강제한다"

#### 명제

한국은 AI 주권 정책 때문에 단일 빅테크 종속이 구조적으로 불가능하고, 이것이 멀티에이전트 환경을 강제한다. 멀티에이전트가 강제되면 에이전트 간 공유 메모리는 선택이 아니라 필수.

#### 한국 특수 상황

1. **한국형 AI 모델 사용 권장/의무화**: 공공기관이 HyperCLOVA X, EXAONE, Solar 등 국산 모델 우선 사용
2. **국산 모델만으로는 부족**: 성능/기능 격차로 글로벌 모델 병행 사용 불가피
3. **결과: 멀티에이전트 환경이 선택이 아니라 강제** — 하나의 빅테크에 올인할 수 없는 구조

#### 규제가 만드는 필수 요건

| 규제/정책 방향 | 요구사항 | enVector/Rune 대응 |
|---------------|---------|---------------------|
| AI 의사결정 추적/감사 | 에이전트 판단 근거 기록 보존 | capture/recall — 의사결정 맥락 자동 보존 |
| 개인정보보호법 강화 | AI가 다루는 데이터 암호화 | FHE — 서버도 평문 열람 불가 |
| 데이터 주권 | 한국 데이터의 해외 평문 저장 금지 | 암호화 상태 처리, 복호화 키는 국내 Vault |
| 한국형 모델 우선 | 복수 모델 환경 필수 | agent-neutral MCP 기반 |

#### 핵심 메시지

> "한국은 AI 주권 정책 때문에 단일 빅테크에 올인할 수 없다. 이것은 멀티에이전트 환경을 구조적으로 강제하고, 에이전트 간 공유 메모리를 필수재로 만든다. 그리고 그 메모리는 당연히 암호화되어야 한다."

### 4단: 아키텍처 — "공유 메모리 자체가 AI-native여야 한다"

#### 명제

다른 에이전트 개발자들이 RAG 파이프라인으로 '에이전트 메모리'를 만들고 있지만, 이것은 검색 시스템을 메모리라고 부르는 것에 불과하다. AI-native 메모리는 근본적으로 다른 아키텍처를 요구한다.

#### 고전적 접근(RAG)의 구조적 한계

```
문서 → 청킹 → 임베딩 → 벡터DB → 검색 → LLM에 주입
```

| 문제 | 설명 |
|------|------|
| 청킹이 추론 구조를 파괴 | "Redis를 거부했다"와 "PostgreSQL을 선택했다"가 별개 청크 → 인과관계 소멸 |
| 명시적 검색만 가능 | 사용자가 "찾아줘"라고 해야 작동 |
| 파이프라인 유지보수가 영구적 | 청크 사이즈, 오버랩, 인덱스 — 끝나지 않는 운영 부담 |
| 지능이 서버에 있음 | 서버가 평가/필터링/추출 → 별도 LLM API 키 필요, 에이전트 맥락과 단절 |

핵심: RAG는 "문서를 검색하는 시스템"이지, "조직의 추론을 기억하는 시스템"이 아님. 범주 오류.

#### Rune의 AI-native 아키텍처

**1) Agent-Delegated Model — 지능의 위치 역전**

| | 고전적 RAG | Rune |
|--|-----------|------|
| 누가 평가 | 서버 (별도 LLM) | 호출한 에이전트 (대화 맥락 보유) |
| 누가 추출 | 서버 파이프라인 | 에이전트가 structured JSON 추출 |
| 인프라 역할 | 지능 + 저장 | 암호화 + 저장만 (LLM API 키 불필요) |

에이전트가 대화 전체 맥락을 보유하므로 판단 품질이 높고, 에이전트(Claude 등)가 개선되면 capture 품질이 자동 개선.

**2) Semantic Compression — 청킹 대신 의미 보존 분해**

- Phase Chain: 순차적 추론(문제 정의 → 대안 평가 → 결정)을 단계별 분리, `group_id`로 연결 유지
- Bundle Split: 풍부한 분석을 core + facet으로 분리, 구조적 관계 보존

RAG 청킹은 텍스트의 물리적 위치로 자르지만, Rune은 추론의 의미적 구조로 분해.

**3) Proactive Recall — 검색이 아니라 기억**

고전적: 개발자가 "MongoDB 결정 찾아줘" → 검색 → 결과
AI-native: 개발자가 "MongoDB 쓸까?" → 에이전트가 맥락 판단하여 자율 recall → "3월에 팀이 ACID 요건 때문에 PostgreSQL을 선택했습니다"

명시적 검색 명령 없이 에이전트가 "지금 조직 메모리가 관련 있다"고 스스로 판단.

#### Rune vs. 고전적 접근 비교

| 측면 | 고전적 RAG | Rune (AI-native) |
|------|-----------|-----------------|
| 지능의 위치 | 서버 | 에이전트 |
| 추론 구조 | 청킹으로 파괴 | Phase Chain/Bundle로 보존 |
| 데이터 보안 | 평문 (IAM 의존) | FHE 암호화 (수학적 보장) |
| 검색 모델 | 명시적 질의 | 자율적 맥락 기반 recall |
| 에이전트 종속 | 벤더 생태계 종속 | MCP 기반 벤더 무관 |
| 운영 부담 | 영구적 파이프라인 관리 | 플러그인 설치로 완료 |

#### 핵심 메시지

> "다른 에이전트 개발자들이 RAG 파이프라인으로 '에이전트 메모리'를 만들고 있지만, 이것은 문서 검색 시스템에 메모리라는 이름을 붙인 것이다. AI-native 메모리는 에이전트가 스스로 판단하고, 추론 구조를 보존하며, 맥락에 따라 자율적으로 기억을 환기하는 시스템이어야 한다. Rune이 그 아키텍처이며, 이것이 enVector의 기술적 해자(moat)다."

## 전체 프레임 요약

| 단 | 핵심 질문 | 답변 |
|----|----------|------|
| 1단: 패턴 | "정말 기업 도구가 되나?" | IT 역사에서 반복된 구조적 패턴. 공유 메모리가 전환점. |
| 2단: 전환점 | "증거가 있나?" | 빅테크가 동시에 베팅. 애널리스트 데이터가 뒷받침. |
| 3단: 가속기 | "한국에서도?" | 규제가 멀티에이전트를 강제. 공유 메모리는 필수재. |
| 4단: 아키텍처 | "기존 방식으로 하면 안 되나?" | RAG는 범주 오류. AI-native 아키텍처가 필요. Rune이 그것. |

## 사용법

4단을 순서대로 전개하되, 상황에 따라:
- 시간이 짧으면: 1단(30초) + 3단(30초)으로 압축
- 기술 이해도가 높으면: 4단을 깊게
- "한국에서도?"가 핵심이면: 3단을 앞으로

## 예상 반론과 대응

| 반론 | 대응 |
|------|------|
| "이번엔 다를 수 있다" | 빅테크 베팅 + 애널리스트 수치로 2단 전개 |
